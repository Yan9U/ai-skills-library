# Questionnaire Guide

Use this guide when the project is blocked or partially blocked by missing client information.

## Goal

Produce a short client-facing document that collects the minimum information required to scope the work correctly. Keep it practical, polite, and specific.

## Default Language

Unless the user asks otherwise, write the questionnaire in Simplified Chinese because the target clients are Chinese-speaking.

## Required Sections

Include only the sections that matter for the project. Start with the highest-risk blockers.

### 1. Project Goal

Clarify the final deliverable and success criteria.

Good questions:

- What exact output should the final project produce?
- What problem should the deliverable solve?
- How will the client decide the project is complete and acceptable?

### 2. Inputs And Existing Materials

Collect the raw materials needed to begin.

Good questions:

- Is there existing source code, data, spreadsheets, images, or documentation?
- Can the client share sample input files and expected outputs?
- Are there target APIs, websites, databases, or folders that must be used?

### 3. Technical Environment

Use this section when language, platform, or environment details matter.

Good questions:

- Which language, platform, or software must be used?
- Which operating system or machine will run the final solution?
- Are there version constraints for Python, MATLAB, Java, libraries, or toolboxes?

### 4. Access, Credentials, And Permissions

Use this section when the project depends on external systems.

Good questions:

- Are login credentials, API keys, or database access required?
- Does the client have permission to access or automate the target system?
- Can the client provide a safe test account or sandbox environment?

### 5. Hardware Or Device Requirements

Use this section for embedded, robotics, camera, audio, networking, or specialized compute work.

Good questions:

- Does the project require a specific device, hardware model, or sensor?
- What are the device model numbers and operating conditions?
- Can the client provide logs, screenshots, videos, or measurement samples?

### 6. Acceptance Criteria

Make the work testable.

Good questions:

- What exact result should be demonstrated at delivery time?
- Are there performance targets, accuracy targets, or runtime limits?
- What test cases should the solution pass?

### 7. Timeline And Budget

Expose deadline risk early.

Good questions:

- What is the target deadline?
- Is there any flexibility in scope or timing?
- Is there a preferred budget range?

## Writing Rules

- Use Simplified Chinese by default.
- Prefer 8 to 18 targeted questions over a huge questionnaire.
- Group questions by topic.
- Put blockers first.
- Do not ask for information that can already be inferred from the brief.
- If one question depends on another answer, mention the dependency briefly.
